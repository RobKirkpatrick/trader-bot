strategy.py (63 lines)

  - Position sizing and risk parameters

models.py (169 lines)
  - WeatherPosition: Open/closed position tracking
    Fields: market_ticker, city, weather_type, threshold, direction,
            nws_probability, entry_price, edge_at_entry, contracts, pnl
  - MarketOpportunity: Scanner output (opportunities waiting for execution)
    Fields: market_ticker, market_title, nws_probability, edge_yes, edge_no,
            recommended_side, recommended_edge, days_to_resolution
  - NWSForecast: Cached NWS data (precip, temp, snow, wind)
  All dataclasses with to_dict()/from_dict() for DynamoDB serialization.

nws_client.py (339 lines)
  - NWSClient: Async HTTP client for National Weather Service API
    Methods:
      • get_grid_point(lat, lon) → grid_id, grid_x, grid_y
      • get_hourly_forecast(grid_id, x, y) → list of hourly periods
      • get_grid_data(grid_id, x, y) → probabilistic time series
      • get_precip_probability(lat, lon, date) → 0.0-1.0
      • get_temperature_forecast(lat, lon, date) → (high_temp, std_dev)
      • get_snow_probability(lat, lon, date, threshold) → 0.0-1.0
      • compute_temp_exceedance_prob(forecast, std_dev, threshold, direction)
  - fetch_nws_forecast(): High-level function to fetch complete forecast
  - Async/await, rate limiting (1s between requests), in-memory caching
  - No scipy dependency (uses error function approximation for Lambda)

market_parser.py (283 lines)
  - WeatherMarketParser: Parse Kalshi market titles
    Handles 5+ regex patterns for common formats:
      "Will NYC high temperature exceed 80°F on April 10?"
      "Will it rain more than 0.5 inches in Chicago on April 12?"
      "Will Boston receive more than 3 inches of snow on April 15?"
    Fallback to substring matching for robustness.
    Methods:
      • parse_market(market_dict) → {city, weather_type, threshold, direction, target_date}
      • get_nws_probability(parsed, nws_forecast) → 0.0-1.0
  - City validation against CITY_COORDS
  - Date parsing (Month Day → YYYY-MM-DD)

scanner.py (339 lines)
  - WeatherMarketScanner: Search for weather markets, compute edge, store opportunities
    Methods:
      • run() → async main loop
      • _search_weather_markets() → find weather-related Kalshi markets
      • _process_market() → parse, validate, compute NWS edge, check qualifications
      • _store_opportunities() → write to DynamoDB
      • _send_alert() → SNS notification with opportunity summary
  - Handles:
    1. Keyword search (weather, rain, temperature, etc.)
    2. Title parsing (extract city, threshold, resolution date)
    3. City validation (must be in CITY_COORDS)
    4. Resolution date validation (0.5-7 days from now)
    5. Bid-ask spread check (< 10 cents)
    6. NWS forecast fetch and probability calculation
    7. Edge calculation (NWS prob - Kalshi ask)
    8. DynamoDB storage of qualifying opportunities
    9. SNS alert with top opportunities
  - Runs every 4 hours via EventBridge

monitor.py (401 lines)
  - WeatherPositionMonitor: Execute trades and monitor open positions
    Methods:
      • run() → async main loop (Phase A + Phase B)
      • _phase_a_execute() → execute new trades
        1. Fetch open opportunities from DynamoDB
        2. Check position count and bankroll limits
        3. Re-fetch NWS to confirm edge
        4. Calculate position size (scales with edge strength)
        5. Place Kalshi limit order
        6. Write WeatherPosition to DynamoDB
        7. Send SNS execution alert
      • _phase_b_monitor() → monitor open positions
        1. Check if market has resolved
        2. Re-fetch NWS forecast
        3. Exit early if NWS reversed >15 cents against position
        4. Otherwise hold to resolution
      • _place_kalshi_order() → wrapper for Kalshi client
      • _close_position() → update DynamoDB when position closes
  - Handles position execution, monitoring, early exit logic
  - Runs every 6 hours via EventBridge

KEY DESIGN DECISIONS
====================

1. NO SCIPY DEPENDENCY
   - Uses error function approximation (Abramowitz-Stegun) for temperature probabilities
   - Keeps Lambda deployment package small and fast
   - Full compatibility with Lambda's Python environment

2. ASYNC/AWAIT THROUGHOUT
   - Efficient parallel NWS requests for multiple cities
   - Scanner can process 50+ markets concurrently
   - Monitor can check 8+ positions concurrently

3. AGGRESSIVE CACHING
   - NWS forecasts cached 1 hour (NWS updates hourly)
   - Grid points cached in-memory during execution
   - Optional DynamoDB cache for multi-invocation persistence

4. EARLY EXIT ON NWS REVERSAL
   - If NWS forecast shifts >15 cents against position, exit early
   - Avoids holding bad bets on weather that changed
   - Keeps position close to expected value

5. POSITION SIZE SCALES WITH EDGE
   - Stronger edges (>$0.20) get larger positions
   - Weaker edges ($0.10-$0.15) get smaller positions
   - Reduces capital at risk when edge is marginal

6. REUSES EXISTING KALSHI CLIENT
   - Imports carpet_bagger.kalshi_client directly
   - No code duplication
   - Benefits from existing RSA signing and auth

7. DynamoDB FOR STATE
   - Opportunities written by scanner, read by monitor
   - Positions written by monitor, read for P&L tracking
   - Supports multi-Lambda architecture (scanner ≠ monitor invocation)
